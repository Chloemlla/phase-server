# Phase - 需求文档

## 1. 项目概述

### 1.1 项目名称

**Phase** （取自"相位/阶段"，暗示时间周期——TOTP 的本质）

### 1.2 一句话描述

自托管的端到端加密 2FA 令牌管理器，基于 Cloudflare Workers 免费计划，Tauri 2.0 全平台客户端。

### 1.3 项目背景

现有的 2FA 管理方案（Google Authenticator、Authy、1Password 等）存在以下问题：

- **Google Authenticator**: 无云同步（或需要 Google 账号），不加密，无法跨设备
- **Authy**: 依赖 Twilio 服务器，闭源，无法自托管
- **1Password/BitWarden**: 功能过重，2FA 只是附属功能，付费或自托管成本高

Phase 旨在提供一个**专注于 2FA 管理**的轻量级解决方案，核心特点是：

- 完全自托管，数据完全由用户掌控
- 零成本运维（Cloudflare 免费计划）
- 端到端加密，服务端零明文
- 一键部署，无需运维知识

### 1.4 仓库结构

| 仓库 | 职责 | 技术栈 |
|------|------|--------|
| `phase-server` | 后端 API 服务 | Cloudflare Workers + D1 (SQLite) + Bun |
| `phase-client` | 全平台客户端 | Tauri 2.0 + React + TypeScript |

## 2. 目标用户

### 2.1 主要用户

- 有自部署能力或意愿的技术用户
- 对数据隐私有较高要求的个人用户
- 希望零成本运维 2FA 管理的用户

### 2.2 用户场景

1. **新用户**: Fork 仓库 → 点击 Deploy to Cloudflare → 配置主密码 → 开始使用
2. **迁移用户**: 从 Google Authenticator / Aegis / andOTP 导入已有令牌
3. **日常使用**: 打开客户端 → 生物识别解锁 → 查看/复制 OTP 代码
4. **多设备**: 在手机上添加令牌 → 自动同步到桌面端

## 3. 功能需求

### 3.1 核心功能（P0 - 必须实现）

#### 3.1.1 用户认证

- **注册**: 设置主密码，客户端从主密码派生加密密钥和认证密钥
- **登录**: 使用主密码认证，服务端验证认证密钥（非加密密钥）
- **会话管理**: JWT Token，支持多设备同时登录，支持撤销会话

#### 3.1.2 令牌管理

- **添加令牌**:
  - 扫描 QR 码（解析 otpauth:// URI）
  - 手动输入 secret + 参数
- **查看令牌**: 列表展示所有令牌，实时生成 OTP 代码
- **编辑令牌**: 修改服务名称、账户名、图标、分组
- **删除令牌**: 确认后删除，支持回收站（30天内可恢复）
- **排序**: 支持拖拽排序、按名称排序、按使用频率排序

#### 3.1.3 OTP 生成

- **TOTP**: RFC 6238 标准实现
  - 支持 SHA-1/SHA-256/SHA-512 算法
  - 支持 6/7/8 位数字
  - 支持自定义时间步长（默认 30 秒）
- **一键复制**: 点击令牌卡片直接复制当前 OTP 到剪贴板
- **倒计时**: 圆形进度条显示当前 OTP 剩余有效时间

#### 3.1.4 端到端加密

- 主密码通过 PBKDF2（或 Argon2，视 WebCrypto 支持情况）派生密钥
- 所有令牌数据在客户端使用 AES-256-GCM 加密后再上传
- 服务端只存储加密后的 vault 数据，无法解密
- 加密密钥永远不离开客户端

#### 3.1.5 多设备同步

- 客户端本地维护 vault 副本
- 启动时与服务端同步（基于版本号）
- 修改后自动推送到服务端
- 冲突处理：基于版本号的乐观锁，冲突时由客户端合并

### 3.2 重要功能（P1 - 应该实现）

#### 3.2.1 生物识别解锁

- 支持指纹 / 面容 / Windows Hello 解锁
- 首次登录需输入主密码，之后可用生物识别替代
- 通过 Tauri 的原生系统密钥链安全存储派生密钥

#### 3.2.2 令牌分组与搜索

- 自定义分组（如：工作、个人、金融）
- 全文搜索（服务名、账户名）
- 收藏/置顶功能

#### 3.2.3 数据导入

- 从 Google Authenticator 导入（扫描迁移二维码）
- 从 Aegis 导入（JSON/加密 JSON 格式）
- 从 andOTP 导入（JSON/加密 JSON 格式）
- 从 2FAS 导入
- 通用 otpauth:// URI 导入

#### 3.2.4 加密备份与恢复

- 导出加密备份文件（.phase 格式）
- 备份文件使用独立密码加密（非主密码）
- 从备份文件恢复到新实例

### 3.3 未来功能（P2 - 后续考虑）

- HOTP 支持（RFC 4226）
- Steam Guard 令牌支持
- 浏览器扩展（自动填充 2FA 代码）
- 暗色/亮色主题切换
- 多语言支持（i18n）
- WebAuthn / Passkey 作为替代认证方式

## 4. 非功能需求

### 4.1 安全性

- 服务端零明文：所有敏感数据在客户端加密后存储
- 主密码不传输：认证使用派生后的认证密钥
- HTTPS 强制：所有 API 通信必须 TLS 加密（Cloudflare 默认提供）
- 防暴力破解：登录失败限速（基于 IP）
- 会话安全：JWT Token 有过期时间，支持主动撤销

### 4.2 性能

- API 响应时间 < 200ms（p95）
- 客户端 OTP 生成延迟 < 10ms
- 客户端冷启动时间 < 2 秒
- Vault 同步（100 个令牌）< 1 秒

### 4.3 可靠性

- 离线可用：客户端本地缓存完整 vault，无网络也能生成 OTP
- 数据持久性：依赖 D1 的持久性保证
- 备份恢复：加密备份文件可在完全新的实例上恢复

### 4.4 部署约束

- **Cloudflare 免费计划限制**:
  - Workers: 100,000 请求/天，10ms CPU 时间/请求
  - D1: 5M 行读取/天，100K 行写入/天，5GB 存储
  - 个人使用场景下这些限制完全足够
- **一键部署**: CF Deploy Button，用户 Fork 后点击按钮即可部署
- **运行时**: Bun（开发和构建工具链）

### 4.5 兼容性

- **桌面**: Windows 10+, macOS 12+, Linux (主流发行版)
- **移动**: iOS 16+, Android 10+（Tauri 2.0 支持）
- **浏览器引擎**: WebView2 (Windows), WebKit (macOS/iOS), WebKitGTK (Linux)

## 5. 用户界面需求

### 5.1 主界面

- 令牌列表：卡片式布局，每张卡片显示：服务图标、服务名称、账户名、当前 OTP、倒计时进度条
- 分组标签栏：顶部可左右滚动的分组标签，"全部" + 用户自定义分组
- 搜索栏：支持按服务名、账户名搜索
- 添加按钮：浮动操作按钮（FAB），支持扫码和手动输入

### 5.2 令牌卡片交互

- 点击卡片：复制当前 OTP 到剪贴板，显示"已复制"提示
- 长按/右键：显示操作菜单（编辑、删除、移动分组、置顶）
- 拖拽：支持拖拽排序

### 5.3 设置页面

- 服务端 URL 配置
- 修改主密码
- 生物识别开关
- 会话管理（查看/撤销其他设备）
- 数据导入/导出
- 自动锁定时间设置
- 关于 / 版本信息

### 5.4 客户端 UI 风格

- 现代简洁设计，参考 Aegis Authenticator 的简洁风格
- 暗色模式为默认（安全类应用的惯例）
- 关键色调：深灰/黑色背景 + 蓝色强调色
- 平台原生感：遵循各平台设计规范（圆角、间距等）

## 7. 术语表

| 术语 | 定义 |
|------|------|
| **Vault** | 所有令牌数据的加密集合，作为一个整体进行加密和同步 |
| **Master Password** | 用户设置的主密码，用于派生加密密钥和认证密钥 |
| **Master Key** | 从主密码通过 PBKDF2 派生的中间密钥 |
| **Encryption Key** | 从 Master Key 通过 HKDF 派生，用于 AES-256-GCM 加密 vault |
| **Auth Hash** | 从 Master Key 派生的认证哈希，发送给服务端做登录验证 |
| **TOTP** | Time-based One-Time Password，基于时间的一次性密码 (RFC 6238) |
| **HOTP** | HMAC-based One-Time Password，基于计数器的一次性密码 (RFC 4226) |
| **E2E** | End-to-End Encryption，端到端加密 |
| **D1** | Cloudflare 的 SQLite 数据库服务 |
